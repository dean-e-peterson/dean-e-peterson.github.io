Disability and Accessibility Contents

When people think about web accessibility, they often focus on people who have difficulty seeing or hearing the content. While these are completely valid concerns, people with other conditions can also face challenges using web content:

  • Autism and Accessibility
  • Hand Tremors and Accessibility
  • P.O.U.R. Accessibility Mnemonic

Autism and Accessibility

Autism Description

Autism is a spectrum of related conditions which cause problems with social skills, repetitive behaviors, issues with speech and communication, etc. (from What is Autism, by Autism Speaks).

Autism Web Accessibility Tips

Web developers can help users with autism by doing the following:

  • Have a clear, uncluttered site.
  • Avoid things that distract people. Specifically...
    • Avoid animation and movement, or have a way to stop them
    • Avoid background music and sounds
    • Do not use a lot of bright primary colors
  • Make your site navigation act consistently
  • Use clear, plain, literal language
  • Avoid metaphors, figures of speech, and any language that is figurative or can have more than one meaning
  • Use bullet lists instead of a wall of text
  • Provide visual alternatives to text information

The suggestions above may help all users, but they are especially important for users with autism.

Autism Accommodations

I did not see very many products to help people with autism use the web. One user found Site-Specific Browsers (SSBs) useful (Jamie Knight interview). These make a specific web page act like an application, without the full web browser user interface, which can simplify the experience. An example is a Mac application called Fluid. Apps from the Apple, Google, and Windows app stores that target a specific web platform (like Facebook) may have similar benefits, at least when compared with accessing those same platforms using a normal web browser.

Hand Tremors and Accessibility

Tremor Description

Hand tremors are an unintentional, continual shaking of the hands. They can grow worse as we age. They are sometimes the result of Parkinson's disease or a stroke. Tremors can make it difficult to use a mouse, a touch screen, or even a keyboard.

Tremor Web Accessibility Tips

Web developers can help users with tremors by doing the following:

  • Make clickable areas like links large.
  • Avoid tiny font sizes.
  • Make sure your site reacts well to being zoomed in. (Some users with tremors and or eye problems may zoom their web browser in to help handle those issues. However, if your site does not zoom well, the user might have to scroll horizontally, negating some of the benefit.)

Tremor Accommodations

People with hand tremors may want to consider the following accommodations.

  • Operating System accessibility settings such as:
    • Filter Keys
    • Sticky Keys
    • Mouse speed settings
  • Products such as:
    • Voice recognition software
    • Specialty Computer Mice
    • Keyguards - They go over the keyboard. The user rests their hands on the keyguard, which has holes in it that the user can type through with fewer accidental keystrokes due to tremors.

P.O.U.R. Accessibility Mnemonic

P.O.U.R. is an accessibility mnemonic standing for Perceivable, Operable, Understandable, and Robust. It is also the organizing principal of the W3C's WCAG 2.0 Web Content Accessibility Guidelines .

Perceivable

All web users should be able to perceive all of the content on the website. The content should be available to multiple senses so that those with visual or hearing impairments can still perceive it.

This site tries to be perceivable by providing alternate text for images, by including "skip navigation" and "to top" links, and by providing page structure with semantic markup and descriptive heading elements.

Operable

All web users should be able to operate the user interface of the website. This includes, but is not limited to, navigation links, search functionality, and forms. Anything the user interacts with should be usable by all users, even those who have difficulty seeing or using a mouse.

This site tries to be operable by using hyperlinks that describe the target without outside context and by labeling all form fields. Navigation menu links are rather large, and are also accessible in logical order via the keyboard.

Understandable

Making a web site understandable to all users involves a few things:

  • Use simple language if possible.
  • Don't surprise the user, especially with unexpected behaviors in places other than where the user is focusing. Keep the context around what the user is doing consistent, beginning with consistent navigation and behavior. As another example, users of screen readers may not see a change you make elsewhere on the page, so only change the state of the whole page when you tell the user that is what you are going to do, like with a submit button.
  • Use semantic markup to help assistive technologies help the user. For example, specifying the language that a page is written in, or specifying the language of a quotation in a different language, will help a screen reader to pronounce the content appropriately, instead of confusingly. Other markup such as the abbreviation tag can help, too.

This site tries to be understandable by specifying the language with the <html> tag lang attribute, by including full versions of appropriate abbreviations with the <abbr> tag, by not changing the context or layout of the site or page except for an user-initiated form submission, and by the use of bullet lists where a list of items may be simpler to understand than prose.

Robust

The website should be well-designed using web standards and best practices. The website should use valid HTML. This helps make the website accessible from both today's technology (some of which you may not have available to test) and tomorrow's technology as well.

This site tries to be robust by using standard, validated HTML and CSS, by attempting to follow best practices in the use of semantic markup and accessibility tools like alternative text, and by at least rudimentary testing on multiple browsers.

Conclusion

These are my interpretations of P.O.U.R. For more authoritative or specific information, see the WCAG 2.0 Web Content Accessibility Guidelines.
